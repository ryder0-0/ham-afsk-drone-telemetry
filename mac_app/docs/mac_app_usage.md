# Using the Mac app with the hardware receiver

This doc covers **connecting the macOS app to a real Bell-202 AFSK receiver
built from `firmware/receiver/`** — the end-to-end chain from drone radio
back to live PFD.

## 1. Cable up the ground station

```
Baofeng UV-5R  ──3.5mm speaker─▶  Audio conditioning  ─▶  ESP32 GPIO34 (ADC)
                                                         │
                                  ESP32 USB-C ──────────▶ Mac (USB)
                                                         │
                                                         └── /dev/cu.usbserial-*
```

1. Flash `firmware/receiver/` to the ESP32 via PlatformIO:

   ```bash
   cd firmware/receiver
   pio run -t upload
   pio device monitor -b 115200   # optional — sanity check boot banner
   ```

2. Plug the Baofeng's speaker out into the audio-conditioning board
   (1 kΩ / 100 nF HPF + DC bias — see `hardware/`).
3. Leave the ESP32 plugged into the Mac over USB.  The Arduino-ESP32 CDC
   driver is built into recent macOS — no kext required.

## 2. Launch the Mac app

From the repo root:

```bash
cd mac_app
swift run -c release
```

Or: open `mac_app/Package.swift` in Xcode 15+ and hit Run.

## 3. Connect

- In the **CONNECTION** sidebar panel, click the refresh icon to rescan
  ports if yours isn't listed.  Pick `/dev/cu.usbserial-*` (or
  `cu.SLAB_USBtoUART` depending on USB-UART chip).
- Leave baud at 115200.
- Click **Connect**.  The dot turns green, and the byte counter starts
  ticking up from `[RX] …` diagnostic lines even when no drone is
  transmitting.

## 4. Get the transmitter talking

Either:

- Flash `firmware/transmitter/` to a second ESP32 wired to a flight
  controller's TELEM2 port, key the Baofeng PTT, and let it transmit
  heartbeats every 5 s; or
- Bench-loopback: feed a WAV generated by `tools/wav_test_generator/` into
  an audio cable from Mac → Baofeng mic (while Baofeng transmits on the
  same frequency your RX is tuned to).

Within a few seconds:

- **Status** panel → `pkt/s` goes positive, `OK` count climbs.
- **Flight** tab → attitude indicator starts updating (if the drone sends
  `ATTITUDE` / `GLOBAL_POSITION_INT`).
- **Map** tab → drone pin + trail appears once GPS fix is 3D.
- **Radio Link** tab → RSSI history plots in real time.

## 5. Interpreting link health

- `Health %` is a heuristic blend of recency + RSSI + CRC pass-rate.
- `RSSI %` is pulled verbatim from the firmware's `rssi=N` diagnostic line,
  which is the quadrature IIR output `mark_pwr / (mark_pwr + space_pwr) ×
  100`.  Values **near 50** mean "no discernible tone" (noise); strong
  signals push towards 0 or 100 depending on which tone is currently on
  the wire.  A noisy but usable link sits around 65–80.
- `CRC fail` spikes → marginal signal; common causes are HT battery sag
  (TX power drop), desense from nearby Wi-Fi, or squelch chopping tones.
- `Seq drops` → packets are getting through CRC-clean but the sequence
  counter jumped, usually meaning one or more previous frames failed CRC
  hard enough to not even parse.

## 6. Logging and replay

Every session automatically writes:

```
~/Library/Application Support/HamTelemetryApp/sessions/<yyyyMMdd-HHmmss>/
  raw.bin           ← every byte received from the ESP32 (append-only)
  snapshots.jsonl   ← one JSON telemetry snapshot per line
```

To replay a past flight:

1. **Logs** tab → **Replay past session** dropdown → pick the folder.
2. Or use the sidebar **REPLAY** panel's *Load…* button and pick any
   `raw.bin` file.

Replay uses a nominal 11520 bytes/sec wall-clock pace (matches 115200 8N1)
adjustable from 0.25× to 8× via the speed slider.

## 7. Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| No port in dropdown | USB-UART driver missing, or cable is charge-only | Install CP210x / CH34x driver; try a different cable |
| Port listed but Connect fails with `errno 16` | Another process has it open (pio monitor?) | `lsof /dev/cu.usbserial-*`, kill the other owner |
| Byte counter rising but `pkt/s = 0` | ASCII diagnostics are arriving but no MAVLink | Check transmitter firmware is flashed and PTT is keying |
| `CRC fail` >> `OK` | Schema mismatch or RF SNR too low | Verify transmitter MAVLink version matches receiver; check audio level |
| Map doesn't pan | Follow toggle off | Toggle **Follow** back on in the map footer |
| Logger "paused" | Storage full / permission denied | Check free space under `~/Library/Application Support/` |

## 8. Alternate input: UDP

If you'd rather route bytes through `mavproxy` or a UDP bridge than connect
directly to the ESP32:

```bash
mavproxy.py --master=/dev/cu.usbserial-0001 --out=udp:127.0.0.1:14550
```

In the app's **Connection** panel pick **UDP**, set port 14550, and click
**Listen**.  Every inbound datagram's payload goes straight into the same
`PacketDecoder` pipeline the serial path uses — so all tabs behave
identically.

## 9. Signed MAVLink v2

Paste your 32-byte signing key (64 hex chars, no spaces, no `0x` prefix)
into **Settings → MAVLink**.  Once valid, signed frames display
`signatureValid = true` in the Logs tab and mismatches bump the `sig fail`
counter.  Leave the key blank to disable verification (frames still pass
through, `signatureValid` is just set to `nil`).

Toggle **Reject replayed timestamps** off during bench testing if you're
replaying the same capture repeatedly.

## 10. Battery profile

**Settings → Battery** picks between the built-in chemistry presets
(2S/3S/4S/6S LiPo, 4S/6S LiHV, 3S/4S/6S Li-ion).  The selected profile's
green/orange/red thresholds are applied immediately to the Power tab's
voltage tile — no restart needed.

## 11. Firmware "quiet" build

If you're running the receiver dedicated to the Mac app and want to
suppress the `[RX] …` ASCII diagnostic lines on USB (pure MAVLink stream,
easier on the edge of link margin):

```ini
# firmware/receiver/platformio.ini
build_flags = -DTELEM_QUIET=1
```

The Mac app still handles the non-quiet default gracefully — this is an
optimisation, not a requirement.

## 12. Exporting to .tlog

**Tools → Export current session → .tlog** (or ⇧⌘E) writes an APM-format
`.tlog` next to the current session's `raw.bin`.  Drag the file into
Mission Planner's **Telemetry Log** viewer, or feed it to `mavlogdump.py`:

```bash
mavlogdump.py --types=GLOBAL_POSITION_INT,ATTITUDE session.tlog
```

Past sessions are exported from the **Logs → Export → .tlog** menu.

## 13. Audio spectrogram (requires firmware extension)

The **Spectrum** tab is wired up on the Mac side to receive
`PKT_TYPE_SPECTRUM = 0x02` packets with a 64-byte payload (32 × `uint16`
magnitude bins + a `uint32` uptime stamp, see
`firmware/common/include/packet.h`).  It renders a scrolling waterfall
with vertical guides at the 1200 Hz MARK and 2200 Hz SPACE bins — an
invaluable diagnostic when the audio chain from radio speaker to ESP32
ADC isn't delivering clean energy at the expected frequencies.

The receiver firmware does **not** yet emit these packets.  When it does,
the tab lights up automatically.  Bin layout is fixed at
`SAMPLE_RATE/2 / 32 = 150 Hz` per bin over 0–4800 Hz.

## 14. Piping MAVLink to Mission Planner / QGC

The Mac app is receive-only.  If you additionally want the decoded frames
in Mission Planner or QGroundControl, keep using the receiver's **UART2
GCS port** (GPIO16 / GPIO17) on a second USB-UART cable — that path emits
clean MAVLink with no diagnostics interleaved and is what `MavlinkOutput`
is designed for.  Run both side-by-side: the Mac app on the USB port, MP
on the UART2 port.
